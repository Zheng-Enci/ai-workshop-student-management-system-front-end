# CLAUDE.md

行为准则，用于减少常见的 LLM 编码错误。可根据项目需要合并使用。

**权衡：** 这些准则倾向于谨慎而非速度。对于简单任务，请自行判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。要明确表达。**

在实现之前：
- 明确陈述你的假设。如果不确定，就去问。
- 如果存在多种解释，都要提出——不要默默选择。
- 如果存在更简单的方法，请指出。在合理的情况下要敢于反对。
- 如果有不清楚的地方，停下来。说出困惑所在。去问。

## 2. 简单优先

**用最少的代码解决问题。不添加任何推测性的内容。**

- 不要添加需求之外的功能。
- 不要为只使用一次的代码创建抽象。
- 不要添加没有要求的"灵活性"或"可配置性"。
- 不要处理不可能出现的情况的错误处理。
- 如果你写了 200 行代码而可以用 50 行实现，那就重写。

问自己："一位资深工程师会说这过度复杂吗？"如果会，就简化。

## 3. 精确修改

**只触碰必须修改的部分。只清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使你用不同方式来做。
- 如果你注意到无关的死代码，提出来——不要删除。

当你的更改产生孤立内容时：
- 移除因你的更改而变得未使用的导入/变量/函数。
- 不要删除原本就存在的死代码，除非被要求。

检验标准：每一行更改都应该直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让测试通过"
- "修复 bug" → "编写能复现问题的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，简要说明计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强的成功标准让你可以独立循环。弱的标准（"让它能工作"）需要不断澄清。

---

**这些准则正在起作用的表现是：**
diff 中不必要的更改更少，因过度复杂而重写的情况更少，以及澄清性问题在错误发生之前提出而不是之后。
