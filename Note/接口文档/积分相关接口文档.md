# 积分管理接口文档

## 文档概述

本文档描述了AI工作坊学生管理系统中所有与积分管理相关的API接口。积分管理模块提供了积分记录创建、积分排名查询等核心功能，支持对学生积分进行灵活的管理和统计。

### 适用范围

- **目标用户**: 系统管理员、学生
- **权限级别**: 部分接口需要特殊密码认证，部分接口公开访问
- **使用场景**: 积分调整、积分记录、积分排名查询

### 安全说明

**重要**: 本文档中的特殊密码等敏感信息均使用占位符表示。实际使用时请：

1. 联系系统管理员获取真实的特殊密码
2. 不要在代码中硬编码敏感信息
3. 使用环境变量或安全的配置管理方式
4. 定期更换特殊密码
5. 限制特殊密码的访问权限

## 基础信息

- **基础URL**: `http://<YOUR_SERVER_IP>:7001`
- **API版本**: v1
- **数据格式**: JSON
- **字符编码**: UTF-8
- **接口文档地址**: `http://<YOUR_SERVER_IP>:7001/doc.html`

## 1. 积分记录接口

### 1.1 创建积分记录

**接口描述**: 通过特殊密码为指定学生创建积分调整记录，支持正负分调整。

**接口地址**: `POST /api/v1/points`

**请求方法**: `POST`

**访问控制**:
- 该接口需要特殊密码认证
- 通过请求头 `X-Special-Password` 传递特殊密码
- 无需Token认证

**请求头**:

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| X-Special-Password | String | 是 | 特殊密码 |
| Content-Type | String | 是 | application/json |

**请求参数** (Request Body):

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| adjustReason | String | 是 | 改分理由，最长500个字符 |
| changePoints | Integer | 是 | 改分分数，支持正负分 |
| targetStudentInfoId | Long | 是 | 目标学生数据库表主键ID |

**请求示例**:

```http
POST /api/v1/points HTTP/1.1
Host: <YOUR_SERVER_IP>:7001
Content-Type: application/json
X-Special-Password: <YOUR_SPECIAL_PASSWORD>

{
  "adjustReason": "参加活动表现优秀",
  "changePoints": 10,
  "targetStudentInfoId": 1
}
```

**响应示例**:

成功响应 (200):
```json
{
  "code": 200,
  "message": "积分记录创建成功",
  "data": null
}
```

失败响应 (400) - 特殊密码错误:
```json
{
  "code": 400,
  "message": "特殊密码错误",
  "data": null
}
```

失败响应 (400) - 参数验证失败:
```json
{
  "code": 400,
  "message": "改分理由不能为空",
  "data": null
}
```

**业务规则**:
- 改分理由不能为空，且长度不能超过500个字符
- 改分分数支持正负分（正数表示加分，负数表示扣分）
- 目标学生ID必须是有效的学生表主键ID
- 每次积分调整都会记录到 `points_record` 表中
- 积分记录包含创建时间，自动记录操作时间

**注意事项**:
- 特殊密码必须正确，否则返回400错误
- `targetStudentInfoId` 是学生表的主键ID，不是学号
- 积分调整会累加到学生的总积分中
- 建议在调整积分时填写清晰的改分理由，便于后续追溯

## 2. 积分查询接口

### 2.1 获取总积分排名前N名

**接口描述**: 根据总积分数降序排列，返回前N名学生的学生表主键ID和总积分数。该接口为公开接口，无需认证。

**接口地址**: `GET /api/v1/points/top-ranking`

**请求方法**: `GET`

**访问控制**:
- 该接口无需Token认证
- 该接口无需特殊密码
- 公开访问接口

**请求参数** (Query Parameters):

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| top | Integer | 否 | 10 | 排名数量，最少为 1 |

**请求示例**:

```http
GET /api/v1/points/top-ranking?top=10 HTTP/1.1
Host: <YOUR_SERVER_IP>:7001
```

**响应示例**:

成功响应 (200):
```json
{
  "code": 200,
  "message": "获取成功",
  "data": [
    {
      "targetStudentInfoId": 1,
      "totalPoints": 150
    },
    {
      "targetStudentInfoId": 5,
      "totalPoints": 120
    },
    {
      "targetStudentInfoId": 3,
      "totalPoints": 100
    }
  ]
}
```

空结果响应 (200):
```json
{
  "code": 200,
  "message": "获取成功",
  "data": []
}
```

失败响应 (400) - 参数验证失败:
```json
{
  "code": 400,
  "message": "排名数量必须大于等于 1",
  "data": null
}
```

**响应字段说明** (TargetStudentInfoIdAndTotalPointsVO):

| 字段名 | 类型 | 说明 |
|--------|------|------|
| targetStudentInfoId | Long | 学生表主键ID（student_info表的主键id） |
| totalPoints | Long | 总积分数（所有积分记录的adjust_points累加和） |

**业务规则**:
- 总积分 = 该学生在 `points_record` 表中所有 `adjust_points` 字段的累加和
- 计算公式：`SUM(adjust_points)`
- 如果学生没有积分记录，总积分为0
- 结果按总积分降序排列（从高到低）
- 如果总积分相同，排序可能不确定（取决于数据库实现）
- 只返回有积分记录的学生（总积分为0的学生不包含在结果中）

**注意事项**:
- `top` 参数最少为 1，如果小于 1 会返回400错误
- 如果不传 `top` 参数，默认返回前10名
- `targetStudentInfoId` 是学生表的主键ID，不是学号
- 该接口只返回学生ID和总积分，不包含学生其他信息
- 如需获取学生详细信息，需要根据 `targetStudentInfoId` 调用学生信息查询接口

**性能说明**:
- 该接口使用数据库聚合查询，性能优化
- 建议合理设置 `top` 参数，避免查询过多数据
- 对于高频访问，可考虑使用Redis缓存优化

### 2.3 获取学生改分总和

**请求示例**:

```http

```

**响应示例**:

成功响应 (200):
```json
{
  "code": 200,
  "message": "获取成功",
  "data": 150
}
```

失败响应 (400) - 参数验证失败:
```json
{
  "code": 400,
  "message": "学生ID不能为空",
  "data": null
}
```

**响应字段说明**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| data | Long | 改分总和（所有积分记录的adjust_points累加和），如果没有记录则返回0 |

**业务规则**:
- 查询条件：基于学生数据库表主键ID（target_student_info_id）查询

**性能说明**:

## 注意事项

1. **特殊密码安全**:
   - 特殊密码必须妥善保管，不要泄露
   - 不要在代码中硬编码特殊密码
   - 建议使用环境变量或配置中心管理

2. **参数验证**:
   - 所有必填参数必须提供
   - 参数格式必须符合要求
   - 参数范围必须在有效范围内

3. **数据一致性**:
   - 积分调整会立即生效
   - 总积分是实时计算的，基于所有积分记录的累加
   - 删除积分记录会影响总积分计算结果

4. **性能优化**:
   - 积分排名查询使用数据库聚合，性能较好
   - 对于高频访问，可考虑使用Redis缓存
   - 建议合理设置查询数量，避免查询过多数据

5. **错误处理**:
   - 所有错误都会返回统一的错误格式
   - 错误信息会明确说明失败原因
   - 建议根据错误码进行相应的错误处理
