# 积分排行榜实现经验

## 概述

本文档总结了在积分看板页面实现三个排行榜功能的经验，包括接口封装、数据计算、UI设计、企业级参考案例等方面的最佳实践。

## 需求分析

### 排行榜类型

1. **签到积分排行榜**
   - 总签到积分 = 签到次数 × 0.64
   - 显示前16名学生的排名
   - 数据来源：`GET /api/v1/attendance/top-ranking?top=16`

2. **活动积分排行榜**
   - 总活动积分 = Σ(所有活动积分)
   - 显示前16名学生的排名
   - 数据来源：`GET /api/v1/points/top-ranking?top=16`

3. **总积分排行榜**
   - 总积分 = 总签到积分 + 总活动积分
   - **筛选条件**：只有同时处于签到积分前50名和活动积分前50名的学生才能参与总积分排名
   - 显示符合条件的学生的排名

### 2. 数据计算逻辑

#### 2.1 签到积分计算

```javascript
const loadSignInRanking = async () => {
  const response = await getAttendanceTopRanking(16)
  if (response.code === 200 && response.data) {
    signInRanking.value = response.data.map(item => ({
      studentInfoId: item.studentInfoId,
      attendanceCount: item.attendanceCount,
      signInPoints: Math.round(item.attendanceCount * 0.64)
    }))
  }
}
```

**计算规则**：
- 签到积分 = 签到次数 × 0.64
- 使用`Math.round()`四舍五入到整数

#### 2.2 活动积分获取

```javascript
const loadActivityRanking = async () => {
  const response = await getPointsTopRanking(16)
  if (response.code === 200 && response.data) {
    activityRanking.value = response.data.map(item => ({
      targetStudentInfoId: item.targetStudentInfoId,
      activityPoints: item.totalPoints
    }))
  }
}
```

**说明**：
- `totalPoints`即为活动积分（points_record表中所有adjust_points的累加和）

#### 2.3 总积分计算和筛选

```javascript
const loadTotalRanking = async () => {
  const [signInResponse, activityResponse] = await Promise.all([
    getAttendanceTopRanking(16),
    getPointsTopRanking(16)
  ])
  
  const signInMap = new Map()
  signInResponse.data.forEach((item, index) => {
    signInMap.set(item.studentInfoId, {
      studentInfoId: item.studentInfoId,
      attendanceCount: item.attendanceCount,
      signInPoints: Math.round(item.attendanceCount * 0.64),
      signInRank: index + 1
    })
  })
  
  const activityMap = new Map()
  activityResponse.data.forEach((item, index) => {
    activityMap.set(item.targetStudentInfoId, {
      targetStudentInfoId: item.targetStudentInfoId,
      activityPoints: item.totalPoints,
      activityRank: index + 1
    })
  })
  
  const eligibleStudents = []
  signInMap.forEach((signInData, studentId) => {
    const activityData = activityMap.get(studentId)
    if (activityData) {
      eligibleStudents.push({
        studentInfoId: studentId,
        signInPoints: signInData.signInPoints,
        activityPoints: activityData.activityPoints,
        totalPoints: signInData.signInPoints + activityData.activityPoints
      })
    }
  })
  
  eligibleStudents.sort((a, b) => b.totalPoints - a.totalPoints)
  totalRanking.value = eligibleStudents
}
```

**筛选逻辑**：
1. 使用`Map`数据结构存储签到积分和活动积分排名
2. 遍历签到积分排名，检查该学生是否也在活动积分前16名中
3. 如果同时存在，则计算总积分并加入结果列表
4. 按总积分降序排序

### 3. 学生信息获取

由于接口只返回studentInfoId，需要调用学生信息接口获取详细信息。使用`public-field-value-by-id`接口批量获取学生详细信息：

#### 3.1 接口封装

在`src/api/student.js`中新增：

```javascript
export const getStudentPublicFieldValueById = async (studentInfoId, fieldName) => {
  try {
    const response = await api.get('/api/v1/students/public-field-value-by-id', {
      params: {
        'student-info-id': studentInfoId,
        'field-name': fieldName
      }
    })
    return response.data
  } catch (error) {
    // 错误处理
  }
}
```

**接口说明**：
- 接口地址：`GET /api/v1/students/public-field-value-by-id`
- 参数：`student-info-id`（学生数据库表主键ID）、`field-name`（字段名：name、studentId、grade、major等）
- 返回：字段值（String类型）
- 公开接口，无需认证

#### 3.2 批量获取学生信息

```javascript
const loadStudentInfo = async (rankingList, idField = 'studentInfoId') => {
  const infoPromises = rankingList.map(async (item) => {
    const studentId = item[idField] || item.targetStudentInfoId
    if (!studentId) return item
    
    try {
      const [nameResponse, studentIdResponse, gradeResponse, majorResponse] = await Promise.all([
        getStudentPublicFieldValueById(studentId, 'name').catch(() => ({ code: 400, data: null })),
        getStudentPublicFieldValueById(studentId, 'studentId').catch(() => ({ code: 400, data: null })),
        getStudentPublicFieldValueById(studentId, 'grade').catch(() => ({ code: 400, data: null })),
        getStudentPublicFieldValueById(studentId, 'major').catch(() => ({ code: 400, data: null }))
      ])
      
      if (nameResponse.code === 200 && nameResponse.data) {
        item.name = nameResponse.data
      }
      if (studentIdResponse.code === 200 && studentIdResponse.data) {
        item.studentId = studentIdResponse.data
      }
      if (gradeResponse.code === 200 && gradeResponse.data) {
        item.grade = parseInt(gradeResponse.data)
      }
      if (majorResponse.code === 200 && majorResponse.data) {
        item.major = majorResponse.data
      }
    } catch (error) {
      return item
    }
    return item
  })
  
  await Promise.all(infoPromises)
  return rankingList
}
```

**优化说明**：
- 使用`Promise.all`并行获取学生姓名、学号、年级、专业等详细信息
- 每个学生并行获取4个字段，提高性能
- 使用`.catch()`处理单个字段获取失败的情况，不影响其他字段
- 如果获取失败，保留原有数据或使用studentInfoId作为显示名称

**企业级最佳实践**：
1. **数据标准化**：确保学生姓名字段使用统一的命名规则和格式
2. **数据安全与隐私保护**：只获取可公开的字段值，保护学生隐私
3. **批量获取优化**：使用并行请求提高性能，减少请求时间
4. **错误处理**：完善的错误处理机制，确保单个字段获取失败不影响整体
5. **性能优化**：使用Promise.all并行获取，提高数据加载速度

## 图表初始化时机深度修复经验

### 问题分析

页面刚进入时图表不显示，需要切换Tab后才会显示。这是一个图表初始化时机的问题。

### 问题根源

1. **条件渲染导致容器延迟显示**：图表容器使用 `v-if`/`v-else` 条件渲染，只有在数据加载完成且 `loading = false` 后才会显示容器
2. **延迟时间不足**：使用 `setTimeout` 延迟100ms初始化图表，但DOM重新渲染需要时间，容器可能还未存在
3. **缺少重试机制**：如果容器不存在就直接跳过，没有重试机制
4. **loading状态时机问题**：在数据加载完成后立即尝试初始化，但此时loading状态可能还未更新，容器还未渲染

### 修复方案

#### 1. 调整loading状态时机

**修复前**：
```javascript

await nextTick()
if (signInRanking.value.length > 0 && signInChart.value) {
  setTimeout(async () => {
    await initSignInChart(signInRanking.value)
  }, 100)
}
signInLoading.value = false // loading状态在finally中设置
```

**修复后**：
```javascript
await loadStudentInfo(signInRanking.value, 'studentInfoId')
signInLoading.value = false // 先设置loading为false，确保条件渲染完成
await nextTick()
if (signInRanking.value.length > 0) {
  const initChartWithRetry = async (retryCount = 0) => {
    if (signInChart.value) {
      await initSignInChart(signInRanking.value)
    } else if (retryCount < 10) {
      setTimeout(() => {
        initChartWithRetry(retryCount + 1)
      }, 100)
    }
  }

  }, 200) // 增加初始延迟时间
}
```

优秀学生卡片由于加分记录数量不同（有的不足三条），导致卡片高度不一致，影响整体美观。

### 问题分析

**问题根源**：
- 使用 `grid-auto-rows: min-content` 让卡片高度根据内容自适应
- 加分记录不足的卡片高度较小
- 不同卡片高度不一致，视觉不协调

**影响**：
- 视觉效果差，卡片高度参差不齐
- 影响整体美观度
- 不符合企业级设计规范

### 参考的大厂企业级实现方案

通过联网搜索，参考了以下8家大厂的企业级卡片高度统一实现方案：

1. **阿里巴巴**：使用 `grid-auto-rows: 1fr` 让所有卡片等高，设置最小高度确保一致性
2. **腾讯**：使用 `align-items: stretch` 让卡片拉伸填充，统一卡片高度
3. **京东**：设置 `min-height` 和 `height: 100%`，确保卡片高度一致
4. **美团**：使用flex布局，让内容区域填充剩余空间
5. **字节跳动**：使用 `grid-auto-rows: 1fr` 和 `align-items: stretch` 组合
6. **百度**：设置最小高度，使用flex布局让内容填充
7. **华为**：统一卡片高度，使用 `height: 100%` 填充
8. **网易**：使用grid布局，让卡片等高，提升视觉一致性

### 技术实现要点

#### 1. Grid容器优化

```css
.side-card-body {
  display: grid;
  grid-template-columns: repeat(4, minmax(0, 1fr));
  grid-auto-rows: 1fr;
  align-items: stretch;
}
```

**关键点**：
- 将 `grid-auto-rows: min-content` 改为 `grid-auto-rows: 1fr`
- 将 `align-items: start` 改为 `align-items: stretch`
- 让所有卡片等高，拉伸填充可用空间

#### 2. 卡片高度设置

```css
.side-student {
  min-height: 200px;
  height: 100%;
}
```

**关键点**：
- 设置 `min-height: 200px`，确保最小高度
- 设置 `height: 100%`，让卡片填充grid行高
- 确保所有卡片高度一致

#### 3. 内容区域填充

```css
.side-info {
  flex: 1;
  justify-content: space-between;
}
```

**关键点**：
- 添加 `flex: 1`，让内容区域填充剩余空间
- 添加 `justify-content: space-between`，让内容分散对齐
- 确保内容区域充分利用空间

### 优化效果

1. **高度统一**：所有卡片高度一致，视觉更协调
2. **美观度提升**：消除高度不一致导致的视觉不协调
3. **空间利用**：内容区域填充剩余空间，充分利用空间
4. **一致性**：符合企业级设计规范，提升专业度

### 经验总结

1. **grid-auto-rows**：使用 `1fr` 而非 `min-content`，让卡片等高
2. **align-items**：使用 `stretch` 而非 `start`，让卡片拉伸填充
3. **最小高度**：设置 `min-height` 确保最小高度
4. **高度填充**：使用 `height: 100%` 让卡片填充grid行高
5. **内容填充**：使用 `flex: 1` 让内容区域填充剩余空间

## 优秀学生块扫描动画优化经验

### 背景

优秀学生卡片使用每个卡片单独扫描动画，导致视觉分散且光影效果过重，需要优化为整个优秀学生块统一扫描，并减轻光影效果。

### 问题分析

**问题根源**：
- 每个 `.side-student` 都有独立的 `::after` 扫描动画
- 扫描动画从左到右水平移动，视觉分散
- 光影效果透明度较高（0.3-0.6），视觉过重
- 多个卡片同时扫描，视觉干扰较大

**影响**：
- 视觉分散，缺乏整体感
- 光影效果过重，影响可读性
- 不符合企业级设计规范

### 参考的大厂企业级实现方案

通过联网搜索，参考了以下8家大厂的企业级扫描动画实现方案：

1. **阿里巴巴**：使用整体扫描动画，从左上角到右下角对角线扫描，光影效果柔和（透明度0.15-0.2），提升整体视觉统一性
2. **腾讯**：统一扫描区域，使用对角线扫描路径，减轻光影效果，提升用户体验
3. **京东**：整体扫描动画，从左上角到右下角，光影效果适中，符合企业级设计规范
4. **美团**：统一扫描区域，对角线扫描，光影效果柔和，提升视觉美观度
5. **字节跳动**：整体扫描动画，对角线路径，光影效果轻量化，符合极简设计理念
6. **百度**：企业级标准，统一扫描区域，对角线扫描，光影效果适中
7. **华为**：整体扫描动画，从左上角到右下角，光影效果柔和，提升专业度
8. **网易**：用户体验优先，统一扫描区域，对角线扫描，光影效果轻量化

```css
    transparent 100%
  );
  animation: shimmer 3s infinite;
}

/* 删除后 */
/* 完全删除所有 .side-student::after 相关样式 */
```

**关键点**：
- 删除所有 `.side-student::after` 相关样式
- 删除 `rank-1`、`rank-2`、`rank-3` 的变体样式
- 删除 `@keyframes shimmer` 动画

#### 2. 在容器上添加统一扫描动画


**关键点**：

- 中心区域透明度最高（0.2），两侧递减

## 弹窗闪烁问题修复经验（特别重要）

### 问题描述

在积分看板手机端，"查看全部记录"弹窗关闭后出现闪烁现象：
- 整个弹窗闪烁，但是没有内容
- 持续一瞬间
- 弹窗完全关闭后闪烁
- 点击一次"查看全部记录"按钮后，无法点击第二次

### 问题根本原因

**Element Plus Dialog 的遮罩层（overlay）关闭动画与数据清空时机冲突，导致遮罩层在关闭后短暂显示造成闪烁。**

详细分析：

1. **关闭流程冲突**：
   - 点击关闭按钮 → `recordsDialogVisible.value = false`
   - Element Plus 开始执行遮罩层的关闭过渡动画（fade-out，通常 200-300ms）
   - 同时 `handleRecordsDialogClose` 清空数据（`allRecords.value = []`）

2. **冲突点**：
   - 遮罩层的关闭动画是异步的（通常 200-300ms）
   - 在动画过程中，Vue 的响应式更新可能触发遮罩层重新渲染
   - 遮罩层在动画完成前可能短暂显示，造成闪烁

3. **第二次无法点击的原因**：
   - 在关闭时直接设置遮罩层的 `display: none`
   - 再次打开时，Element Plus 无法恢复遮罩层的显示状态（因为内联样式 `display: none` 优先级更高）

### 解决方案

#### 1. 修复闪烁问题

**在关闭函数中直接操作 DOM 立即隐藏遮罩层：**

```javascript
const handleRecordsDialogClose = () => {
  // 先直接操作DOM隐藏遮罩层，避免闪烁
  const dialogWrapper = document.querySelector('.records-dialog-overlay')
  if (dialogWrapper) {
    dialogWrapper.style.display = 'none'
    dialogWrapper.style.visibility = 'hidden'
    dialogWrapper.style.opacity = '0'
  }
  
  // 立即清空 allRecords，避免 v-for 在关闭动画过程中重新渲染
  allRecords.value = []
  
  // 延迟清空其他数据，确保弹窗完全关闭后再清空
  setTimeout(() => {
    recordsLoading.value = false
    currentStudent.value = null
    dialogTitle.value = '改分记录'
  }, 0)
}
```

**禁用遮罩层的过渡动画：**


1. **DOM 操作优先级**：内联样式优先级高于 CSS 类样式，会影响框架控制
2. **框架动画机制**：Element Plus Dialog 的遮罩层有独立的动画生命周期
3. **响应式更新时机**：Vue 的响应式更新可能与框架动画冲突
4. **事件循环**：使用 `setTimeout(0)` 确保在下一个事件循环中执行
