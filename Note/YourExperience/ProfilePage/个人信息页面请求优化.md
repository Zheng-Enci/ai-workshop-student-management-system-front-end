# ProfilePage - 个人信息页面请求优化

## 问题描述

个人信息页面进入时，`validation-token` 请求被调用了两次，导致不必要的重复请求。

## 问题分析

### 请求流程

1. **第一次 validation-token 请求**：
   - 用户访问 `/profile` 路由
   - 路由守卫 `beforeEach` 检测到 `meta: { requiresAuth: true }`
   - 调用 `validateToken(token)` 验证 token

2. **设备检测和路由跳转**：
   - `ProfilePage.vue` 组件挂载
   - `onMounted` 钩子调用 `detectDevice()` 检测设备类型
   - 根据屏幕宽度跳转到 `/profile/mobile` 或 `/profile/desktop`

3. **第二次 validation-token 请求**：
   - 路由跳转到 `/profile/desktop` 时
   - 路由守卫再次触发，再次调用 `validateToken(token)`

4. **页面数据请求**：
   - `ProfilePageDesktop.vue` 组件挂载
   - `onMounted` 钩子调用 `loadProfile()`
   - 并行请求 `getStudentProfile(token)` 和 `getMyAttendanceCount(token)`

### 请求列表

根据网络请求记录，个人信息页面进入时执行以下请求：

1. **validation-token** (2次) - 路由守卫验证
2. **profile** (1次) - 获取个人信息
3. **my-attendance-count** (1次) - 获取签到次数

## 优化方案

### 添加 Token 验证缓存机制

在路由守卫中添加短期缓存，避免短时间内重复验证同一个 token。

**位置**：`src/router/index.js`

**实现方案**：

```javascript
let tokenValidationCache = {
  token: null,
  isValid: false,
  timestamp: 0
}

const CACHE_DURATION = 5000

const getCachedValidation = (token) => {
  const now = Date.now()
  if (tokenValidationCache.token === token && 
      (now - tokenValidationCache.timestamp) < CACHE_DURATION) {
    return tokenValidationCache.isValid
  }
  return null
}

const setCachedValidation = (token, isValid) => {
  tokenValidationCache = {
    token,
    isValid,
    timestamp: Date.now()
  }
}
```

**优化后的路由守卫**：

```javascript
router.beforeEach(async (to, from, next) => {
  const token = localStorage.getItem('token');
  
  if (to.meta.requiresAuth) {
    if (!token) {
      next('/login');
      return;
    }
    
    const cachedResult = getCachedValidation(token)
    if (cachedResult !== null) {
      if (!cachedResult) {
        localStorage.removeItem('token');
        localStorage.removeItem('userInfo');
        tokenValidationCache = { token: null, isValid: false, timestamp: 0 }
        next('/login');
        return;
      }
      next();
      return;
    }
    
    try {
      const isValid = await validateToken(token);
      setCachedValidation(token, isValid)
      if (!isValid) {
        localStorage.removeItem('token');
        localStorage.removeItem('userInfo');
        next('/login');
        return;
      }
    } catch (error) {
      setCachedValidation(token, false)
      localStorage.removeItem('token');
      localStorage.removeItem('userInfo');
      next('/login');
      return;
    }
  }
  
  // ... 其他逻辑
});
```

## 优化效果

### 优化前
- `validation-token` 请求：2次（路由守卫触发2次）
- 总请求数：4次

### 优化后
- `validation-token` 请求：1次（第二次使用缓存）
- 总请求数：3次
- **减少 1 次不必要的网络请求**

## 关键要点

### 1. 缓存策略
- **缓存时长**：5秒（5000毫秒）
- **缓存范围**：仅缓存验证结果，不缓存 token 本身
- **缓存失效**：超过缓存时长或 token 变化时自动失效

### 2. 缓存逻辑
- **命中缓存**：如果缓存存在且未过期，直接使用缓存结果
- **未命中缓存**：调用 API 验证，并将结果存入缓存
- **缓存更新**：每次验证后更新缓存

### 3. 安全性
- **短期缓存**：5秒缓存时长，平衡性能和安全性
- **Token 变化检测**：token 变化时自动清除缓存
- **错误处理**：验证失败时清除缓存和本地存储

### 4. 性能优化
- **减少网络请求**：避免短时间内重复验证
- **提升响应速度**：使用缓存时无需等待网络请求
- **降低服务器压力**：减少不必要的验证请求

## 最佳实践

1. **合理设置缓存时长**：
   - 太短：无法有效减少请求
   - 太长：可能影响安全性
   - 建议：5-10秒

2. **缓存失效机制**：
   - 时间过期：超过缓存时长
   - Token 变化：token 更新时清除缓存
   - 验证失败：验证失败时清除缓存

3. **错误处理**：
   - 验证失败时清除所有相关缓存
   - 清除本地存储的 token 和 userInfo
   - 跳转到登录页面

## 总结

通过添加 Token 验证缓存机制，成功减少了个人信息页面进入时的重复请求，提升了页面加载性能和用户体验。缓存机制在保证安全性的前提下，有效减少了不必要的网络请求。
