# Git 提交规范

## 企业级 Git 提交最佳实践

### 参考的企业级实现
通过联网搜索，参考了以下8个企业级实现方案：
1. **Conventional Commits 规范**：使用标准化的提交信息格式，包括类型前缀、范围和描述
2. **类型前缀分类**：使用 `feat`、`fix`、`refactor`、`style`、`docs` 等类型前缀，清晰标识提交类型
3. **提交信息格式**：遵循"类型: 简短描述"的格式，可选的详细描述和脚注
4. **原子性提交**：每次提交只包含一个逻辑变更，便于代码审查和回滚
5. **提交前检查**：提交前检查代码质量、运行测试、确保无编译错误
6. **分支管理策略**：使用功能分支、开发分支、主分支等分支策略
7. **代码审查流程**：提交后通过 Pull Request 进行代码审查
8. **变更日志维护**：提交时同步更新变更日志，记录重要变更

### 提交信息格式

#### 基本格式
```
<type>: <subject>

<body>

<footer>
```

#### 类型前缀（Type）
- **feat**: 新功能
- **fix**: 修复问题
- **refactor**: 代码重构
- **style**: 样式调整（不影响代码逻辑）
- **docs**: 文档更新
- **perf**: 性能优化
- **test**: 测试相关
- **chore**: 构建/工具相关
- **ci**: CI/CD 相关

#### 提交信息示例

**简单提交**
```
feat: 添加用户登录功能
```

**详细提交**
```
feat: 优化补卡表单布局和日期时间快捷选择

- 优化补卡表单布局，使用flex布局让内容区域可滚动，按钮固定在底部
- 将日期快捷选项改为包含日期和时间段的组合选项（今天早上、昨天下午等）
- 选择快捷选项后自动填充日期和时间并切换到时间选择步骤
- 删除快速补卡功能相关代码，简化代码结构
- 更新相关文档和经验总结
```

### 提交前检查清单

1. **代码质量检查**
   - ✅ 代码符合 ESLint 规范
   - ✅ 无编译错误
   - ✅ 无控制台警告

2. **功能验证**
   - ✅ 功能正常工作
   - ✅ 边界情况已处理
   - ✅ 错误处理完善

3. **文档更新**
   - ✅ 更新日志已记录
   - ✅ 经验文档已补充
   - ✅ 相关文档已同步

4. **提交信息**
   - ✅ 提交信息清晰明确
   - ✅ 使用正确的类型前缀
   - ✅ 描述变更内容

### Git 提交命令

#### 基本提交流程
```bash
# 1. 查看状态
git status

# 2. 添加文件到暂存区
git add .

# 3. 提交到本地仓库
git commit -m "feat: 提交信息"

# 4. 推送到远程仓库（可选）
git push
```

#### 使用完整路径的 Git 命令（Windows）
```powershell
# 使用完整路径执行 Git 命令
& "D:\BaiduSyncdisk\ZhengEnCi\RunCode\CodeEnvironment\Git\Git\bin\git.exe" status
& "D:\BaiduSyncdisk\ZhengEnCi\RunCode\CodeEnvironment\Git\Git\bin\git.exe" add .
& "D:\BaiduSyncdisk\ZhengEnCi\RunCode\CodeEnvironment\Git\Git\bin\git.exe" commit -m "提交信息"
```

### 关键要点

- **类型前缀**：使用标准化的类型前缀，清晰标识提交类型
- **简洁描述**：提交信息的第一行要简洁明了，不超过50个字符
- **详细说明**：复杂变更可以在详细描述中说明，使用列表格式
- **原子性提交**：每次提交只包含一个逻辑变更，便于追踪和回滚
- **文档同步**：提交代码时同步更新相关文档
- **提交前检查**：提交前确保代码质量，无编译错误和警告
