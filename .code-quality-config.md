# 代码质量检查工具配置说明

本项目已配置大厂企业级代码质量检查工具，确保代码质量和规范性。

## 一、已配置工具

### 1. Commitlint - 提交信息规范检查

**配置文件**: `commitlint.config.js`

**功能**:
- 检查 Git 提交信息是否符合 Conventional Commits 规范
- 支持的类型: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert, types, wip, release
- 提交信息格式: `<type>(<scope>): <subject>`

**使用方式**:
```bash
# 提交时会自动检查
git commit -m "feat: 添加新功能"

# 手动检查提交信息
npx commitlint --from HEAD~1 --to HEAD --verbose
```

**提交信息示例**:
- ✅ `feat: 添加用户登录功能`
- ✅ `fix(api): 修复接口超时问题`
- ✅ `docs: 更新 README 文档`
- ❌ `update code` (不符合规范)

### 2. Depcheck - 未使用依赖检测

**配置文件**: `.depcheckrc.json`

**功能**:
- 检测项目中未使用的 npm 依赖包
- 检测缺失的依赖包
- 支持 TypeScript、Vue、ES6、JSX 等文件类型

**使用方式**:
```bash
# 检查未使用的依赖
npm run check:deps

# 查看详细报告
npx depcheck --config .depcheckrc.json
```

**配置说明**:
- 忽略构建工具和插件相关的依赖（如 @babel/*, eslint-*, vite-plugin-* 等）
- 忽略配置文件目录
- 支持多种文件类型检测

### 3. Rollup Plugin Visualizer - 打包体积分析

**配置文件**: `vite.config.js`

**功能**:
- 分析打包后的文件体积
- 可视化展示各模块占用空间
- 支持 gzip 和 brotli 压缩大小分析
- 使用 treemap 图表展示

**使用方式**:
```bash
# 构建并生成分析报告
npm run analyze:bundle

# 或使用完整构建流程
npm run build:analyze
```

**输出文件**: `dist/stats.html`

**分析内容**:
- 各模块文件大小
- gzip 压缩后大小
- brotli 压缩后大小
- 模块依赖关系

### 4. npm audit - 依赖安全检查

**功能**:
- 检查项目依赖中的已知安全漏洞
- 提供漏洞修复建议
- 支持自动修复部分漏洞

**使用方式**:
```bash
# 检查安全漏洞（中等及以上级别）
npm run check:security

# 自动修复可修复的漏洞
npm run check:security:fix

# 完整检查（包含所有工具）
npm run check:all
```

**安全级别**:
- low: 低风险
- moderate: 中等风险（默认检查级别）
- high: 高风险
- critical: 严重风险

## 二、Git Hooks 配置

### Pre-commit Hook

**文件**: `.husky/pre-commit`

**功能**:
- 提交前自动运行 lint-staged
- 对暂存文件进行 ESLint、Stylelint 和 Prettier 检查
- 自动修复可修复的问题

**检查内容**:
- JavaScript/Vue 文件: ESLint + Prettier
- CSS/SCSS 文件: Stylelint + Prettier
- JSON/Markdown 文件: Prettier

### Commit-msg Hook

**文件**: `.husky/commit-msg`

**功能**:
- 提交时自动检查提交信息格式
- 不符合规范的提交会被拒绝

## 三、npm 脚本命令

### 代码检查
```bash
npm run check          # 运行 ESLint 和 Stylelint
npm run check:deps     # 检查未使用的依赖
npm run check:security # 检查安全漏洞
npm run check:all      # 运行所有检查
```

### 代码格式化
```bash
npm run format         # 格式化所有文件
npm run format:check   # 检查格式是否符合规范
```

### 构建和分析
```bash
npm run build          # 标准构建
npm run build:analyze  # 构建并生成分析报告
npm run analyze:bundle # 仅生成打包分析报告
```

### 修复命令
```bash
npm run lint:fix       # 自动修复 ESLint 问题
npm run lint:css:fix   # 自动修复 Stylelint 问题
npm run check:security:fix # 自动修复安全漏洞
```

## 四、配置文件说明

### commitlint.config.js
- 提交信息类型枚举
- 提交信息格式规则
- 长度限制和格式要求

### .depcheckrc.json
- 忽略的依赖包模式
- 忽略的文件和目录
- 支持的文件类型检测器

### .lintstagedrc.js
- 不同文件类型的检查规则
- 自动修复配置

### vite.config.js
- 打包分析插件配置
- 分析报告生成设置

## 五、工作流程

### 开发流程
1. 编写代码
2. 提交代码（自动触发 pre-commit hook）
3. 自动检查和修复代码格式
4. 编写提交信息（自动触发 commit-msg hook）
5. 提交信息格式检查

### 构建流程
1. 运行代码检查
2. 运行 CSS 分析
3. 构建项目
4. （可选）生成打包分析报告

### 定期检查
- 每周运行 `npm run check:all` 进行全面检查
- 每月运行 `npm run analyze:bundle` 分析打包体积
- 发现安全漏洞时立即运行 `npm run check:security:fix`

## 六、注意事项

1. **提交信息格式**: 必须符合 Conventional Commits 规范，否则提交会被拒绝
2. **代码格式**: 提交前会自动格式化，确保代码风格一致
3. **依赖管理**: 定期运行 depcheck 清理未使用的依赖
4. **安全漏洞**: 发现漏洞后应及时修复，避免安全风险
5. **打包体积**: 定期分析打包体积，优化项目性能

## 七、故障排除

### Commitlint 检查失败
- 检查提交信息格式是否正确
- 参考提交信息示例修改

### Depcheck 误报
- 检查 `.depcheckrc.json` 中的忽略配置
- 添加需要忽略的依赖模式

### 打包分析报告未生成
- 确保使用 `ANALYZE=true` 环境变量
- 检查 `dist/stats.html` 文件是否生成

### 安全漏洞无法自动修复
- 查看漏洞详情，手动更新依赖版本
- 检查是否有破坏性变更
